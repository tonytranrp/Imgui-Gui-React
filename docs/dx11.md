# DirectX 11 Backend

## Status

`igr_backend_dx11` is the first fully live renderer in the repository.

The backend currently provides:

- D3D11 device and swap-chain bootstrap
- hardware, debug-layer, and WARP fallback logic
- render-target recreation on resize
- Direct2D and DirectWrite interop for text rendering
- caller-registered user textures for image widgets
- clip-rect composition mapped onto DX11 scissor state and clipped Direct2D text
- custom draw primitives through the shared frame model (`fill_rect`, `stroke_rect`, and `line`)
- shared font and image resource registration
- host rebind and detach flows for injected and external-host usage
- per-frame backend stats for overlay instrumentation
- theme-driven quad rendering for the core widget set
- font-aware layout sizing and updated widget chrome aligned more closely with the DX12 sample surface
- runtime validation through native tests and the Win32 DX11 sample

## Lifecycle

The DX11 backend supports two practical lifecycles.

### Owned window

1. Construct `Dx11Backend` with a valid `HWND`
2. Call `initialize()`
3. Build a `FrameDocument` each frame
4. Call `render(document)`
5. Call `present()`
6. Forward `WM_SIZE` to `resize()`
7. Call `shutdown()` during teardown

### External swap chain or injected overlay

1. Construct `Dx11Backend` with caller-provided device, context, and swap chain
2. Set `host_mode` to `external_swap_chain` or `injected_overlay`
3. Set `presentation_mode` to `host_managed`
4. Set `resize_mode` to `host_managed`
5. Set `clear_target` to `false` when compositing onto a game back buffer
6. Leave `restore_host_state` enabled
7. Call `invalidate_back_buffer_resources()` before the host resizes buffers
8. Call `render(document)` from the host-controlled frame hook

## Host-managed and injected usage

DX11 is no longer limited to the sample's app-owned window model.

The backend now supports:

- attaching to an external `IDXGISwapChain`
- deriving the device and immediate context from that swap chain
- host-managed `present()` mode for hooked `Present` flows
- `invalidate_back_buffer_resources()` before host `ResizeBuffers`
- render-state restoration for safer overlay rendering inside host applications
- `rebind_host()` for swap-chain replacement and late host attachment
- `detach_host()` for overlay recovery and teardown without destroying the backend object

This is the current foundation for DLL injection and overlay-style usage where the library user does not own the game window or presentation loop.

For overlays, the intended baseline config is:

- `host_mode = injected_overlay`
- `presentation_mode = host_managed`
- `resize_mode = host_managed`
- `clear_target = false`
- `restore_host_state = true`
- `input_mode = external_forwarded` or `none`

For pointer capture in injected or manual-host flows, use [interaction.hpp](/C:/Users/tonyi/OneDrive/Documents/GitHub/Imgui-Gui-React/include/igr/interaction.hpp). The current helper evaluates the rendered frame document and reports whether the overlay wants mouse or keyboard ownership at the current pointer position.

## Widget Coverage

The DX11 backend currently renders:

- window
- stack
- clip rect containers
- text
- button
- checkbox
- image
- progress bar
- separator
- custom draw primitives (`fill_rect`, `stroke_rect`, `line`)

The renderer is intentionally document-driven. Native C++ and declarative bridge inputs both resolve into the same `FrameDocument`, then flow through the same DX11 pass.

## Font and Text Resources

Text is rendered through DirectWrite on top of the D3D11 back buffer via a Direct2D DXGI surface render target. The backend now also exposes caller-driven font registration through `register_font()` and `unregister_font()`, so text widgets can reference named font resources instead of relying only on the default body/title formats.

Important implications:

- swap-chain buffers use BGRA format for Direct2D interop
- injected external D3D11 devices must support BGRA surfaces for text rendering
- `D2DERR_RECREATE_TARGET` triggers render-target rebuilding logic

## User textures

DX11 now exposes a caller-driven user-texture path through `register_texture()` and `unregister_texture()`. That gives the native runtime and the React bridge a real `image` widget path without forcing the backend to own the host application's asset pipeline.

The current DX11 image model is intentionally explicit:

- the caller registers a named `ID3D11ShaderResourceView`
- the caller can optionally register a named image resource that resolves to that texture plus UV/tint metadata
- the frame model refers to either a texture key or an image resource key
- the backend resolves the key at draw time

This is the right direction for both embedded apps and injected overlays, because the host keeps resource creation under explicit control.

The DX11 API now also exposes `Dx11TextureBinding`, `register_image()`, `register_font()`, `has_texture()`, `has_image()`, and `has_font()` so the resource-registration path is closer to the explicit DX12 texture contract while still using native D3D11 objects.

## Overlay Instrumentation

`frame_stats()` now reports widget counts, quad counts, draw-batch counts, textured batch counts, clip-rect usage, custom-draw usage, and basic CPU-side scene/render timing. That gives injected overlays a first pass at runtime instrumentation without needing external profiling hooks first.

## Clipping

`clip_rect` is now a real renderer primitive, not only a document node. The DX11 path resolves clip containers into:

- `RSSetScissorRects()` for quad and image batches
- `PushAxisAlignedClip()` for Direct2D text draws

That gives the current layout model a practical isolation primitive for nested panels, previews, and future scroll regions without forking the renderer between native and declarative usage.

## Custom draw primitives

The shared frame model now supports:

- `fill_rect()`
- `stroke_rect()`
- `draw_line()`

Those primitives render through the same DX11 batch path as the built-in widgets, so downstream code can add accents, separators, graph rules, and overlay diagnostics without forking the backend.

## Theming

`Dx11BackendConfig` accepts a `Dx11Theme` with color and spacing controls for:

- window chrome
- panels
- buttons
- checkbox states
- progress bars
- separators
- body and title text sizing

## Current Boundaries

DX11 is no longer only a host-owned sample renderer. It now has an explicit manual-host path suitable for engine integrations and injected overlays, but it still needs richer per-frame binding abstractions if we later want to support arbitrary externally bound render targets instead of swap-chain-derived targets.
