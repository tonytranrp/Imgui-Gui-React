# DirectX 12 Backend

## Status

DX12 is now a live renderer, not only a contract shell. The backend supports:

- owned-window rendering with device, queue, swap chain, descriptor heaps, frame resources, fences, and presentation
- owned-window DirectWrite text rendering through a D3D11On12 plus Direct2D interop layer
- host-managed and injected rendering through explicit `Dx12FrameBinding`
- descriptor-backed user textures for `image` widgets
- shared font and image resource registration
- host rebind and detach flows for hostile host runtimes
- per-frame backend stats for overlay instrumentation
- shared custom draw primitives (`fill_rect`, `stroke_rect`, `line`)
- runtime validation through both the DX12 sample and the DX12 native test binaries

## Non-negotiable API criteria

DX12 cannot be treated like DX11 with a few renamed objects. For owned-window apps and injected overlays alike, the backend must keep explicit control over:

- device ownership
- command queue ownership
- command allocator and command list ownership
- descriptor heap ownership
- back-buffer resource ownership
- fence and frame-resource synchronization
- present policy
- resize policy

## Required manual-passing model

For host-managed and injected modes, the caller should be able to provide:

- `ID3D12Device*`
- `ID3D12CommandQueue*`
- `ID3D12GraphicsCommandList*`
- `IDXGISwapChain3*`
- the active back-buffer index
- the active render-target resource or RTV handle
- the descriptor heap used for UI-visible resources
- the user-texture descriptor strategy for React and immediate-mode image widgets

The backend should not guess these values or invent hidden ownership when the host already controls them.

## Current frame binding contract

The backend now has an explicit `Dx12FrameBinding` contract in code for host-managed and injected modes. It carries:

- frame index
- back-buffer index
- command list
- render-target resource
- RTV handle
- shader-visible SRV heap
- whether the host already bound render targets
- whether the host already bound descriptor heaps
- whether the host already transitioned the target
- the expected before and after render-target states
- whether the host wants the backend to clear

`render()` now rejects host-managed DX12 calls until a valid frame binding is provided through `bind_frame()`. That is intentional. A DX12 renderer without an explicit command list, target, and descriptor heap contract would be wrong for overlays and engine integrations.

## Texture binding contract

DX12 image widgets now also have an explicit descriptor-backed registration path through `register_texture()`. The current contract requires:

- a descriptor heap pointer
- a CPU descriptor handle
- a GPU descriptor handle

During `render()`, image widgets are validated against the registered texture table, and the active texture heap must match the shader-visible SRV heap from the bound frame. That keeps the future React and immediate-mode image path aligned with real DX12 descriptor ownership instead of pretending the backend can infer it safely.

That design is the correct fit for injectors, embedded tools, and engine integrations.

The higher-level image registry now lets callers register logical image resources separately from descriptor-backed texture bindings. The frame document stays backend-agnostic while the backend still validates the real DX12 descriptor ownership underneath.

## Descriptor-backed images

DX12 image widgets now use the same live draw path as the rest of the frame document.

The backend exposes:

- `register_texture()`
- `unregister_texture()`
- heap-aware validation for descriptor-backed image bindings

The rule remains explicit: a document `image` key must resolve to a registered `Dx12TextureBinding`, and that binding must belong to the active frame's shader-visible SRV heap.

## Injected overlay criteria

For an injected DX12 overlay:

- the library must not call `Present()` when the host owns presentation
- the library must not resize the swap chain when the host owns resize
- command recording must happen on host-provided objects or clearly caller-owned resources
- resource transitions must be explicit
- descriptor heap use must be predictable and documented

For detach and host rebinding, the host should treat `detach_host()` and `rebind_host()` as frame-boundary operations. In practice that means closing or resetting the active command list before tearing down or replacing the bound DX12 host objects.

For input forwarding in manual-host or injected usage, pair the backend with [interaction.hpp](/C:/Users/tonyi/OneDrive/Documents/GitHub/Imgui-Gui-React/include/igr/interaction.hpp). That keeps overlay capture policy explicit instead of baking Win32 ownership assumptions into the renderer.

## Current renderer shape

The current DX12 draw path includes:

- root signature and PSO creation
- descriptor heap strategy for user textures
- transient upload-buffer-backed vertex streaming
- D3D11On12-backed DirectWrite label rendering for owned-window mode
- explicit resource transition policy around the caller-provided render target
- frame-resource and fence ownership for app-owned mode
- backend frame stats for runtime instrumentation
- runtime validation with a dedicated DX12 sample and contract-plus-render tests

## Remaining work

The highest-value remaining DX12 work is now:

- explicit host-managed text composition beyond the current owned-window D3D11On12 path
- font atlas and longer-term descriptor allocation strategy for overlay-first runtimes
- deeper overlay performance review and instrumentation
